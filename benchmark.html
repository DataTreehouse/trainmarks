Key Observations

Patterns that emerged from the benchmark results.

maplib dominates across the board

With parallel reads and streaming queries, maplib leads on every operation at every scale. At 10M triples: Turtle read in 3.4s (Jena: 19s, RDF4J: 25s), Q2 aggregation in 20ms, that is 43x faster than RDF4J and 2268x faster than rdflib.

The gap widens at scale

maplib's advantage grows with dataset size. Q2 speedup vs RDF4J: 6x at 100K, 16x at 1M, 43x at 10M. The columnar engine (Polars/Arrow) shines on aggregation-heavy queries where row-oriented stores struggle.

rdflib hits a wall at 10M

233s to read Turtle, 63s for Q4. At 10M triples rdflib is 68x slower than maplib on reads and 1283x slower on aggregation queries. Pure Python with no native acceleration simply can't keep up.

QLever: built for SPARQL at scale

QLever delivers the fastest SPARQL across the board at 10M triples: Q1 in 1.3ms, Q2 in 1.8ms, Q4 in 2.3ms. Its on-disk index means slower loading (7.8s for Turtle), but once indexed, query times barely grow with dataset size — Q1 actually gets faster from 1M to 10M.

Methodology

All benchmarks use the same data files and the same SPARQL queries. 5-minute timeout per operation.

Python: maplib, maplib (disk), oxigraph, rdflib

time.perf_counter(), garbage collected between runs. Queries: best of 3 after warmup. maplib uses parallel=True for reads and streaming=True for queries. maplib (disk) uses Model(storage_folder=...) for disk-backed storage instead of in-memory — same API, data persisted to local SSD. oxigraph uses pyoxigraph's Store() without a path argument (anonymous temporary store backed by RocksDB).

Java: Jena, RDF4J

System.nanoTime(), JVM warmed up with a full medium-dataset read before timing. Queries: best of 3 after warmup.

Rust: Kolibrie

Kolibrie (KU Leuven Stream Intelligence Lab) compiled natively on macOS. Dictionary-encoded BTreeSet storage with Streamertail volcano-style optimizer. std::time::Instant for timing. Queries: best of 3 after warmup.

C++: QLever (Docker)

QLever via Docker (adfreiburg/qlever). Index building via qlever-index, queries via HTTP endpoint on qlever-server. Best of 3 after warmup. Write operations are N/A (query engine).

C: Virtuoso (Docker)

Virtuoso Open-Source 7.x via Docker. Bulk loading via isql ld_dir() + rdf_loader_run(). SPARQL queries via HTTP endpoint. Best of 3 after warmup. Write operations are N/A (database server).

Java: GraphDB (Docker)

Ontotext GraphDB 10.8.0 via Docker. Server-side import (files mounted at /root/graphdb-import). Repository with empty ruleset (no inference). SPARQL queries via HTTP endpoint. Best of 3 after warmup. Write operations are N/A (database server).

C#: dotNetRDF (Docker)

dotNetRDF 3.5.1 on .NET 8 via Docker. In-memory TripleStore with TurtleParser/NTriplesParser for I/O and Leviathan SPARQL engine for queries. Best of 3 after warmup. 16 GB container memory limit. xlarge (10M) exceeds available memory (OOM).

Java: Neo4j + neosemantics (Docker)

Neo4j 5.26 Community Edition + neosemantics (n10s) 5.26.0 via Docker. RDF imported via n10s.rdf.import.fetch() into Neo4j's labeled property graph. SPARQL queries translated to equivalent Cypher. Best of 3 after warmup. Write operations are N/A (database server). Note: Neo4j is a property graph database — n10s maps RDF triples to its native model, so import times include the RDF-to-property-graph conversion.

I/O operations

Single timed run (no averaging) since they include allocation overhead that is part of the real-world cost.

Machine

Apple M3 Max, 36 GB RAM, macOS. maplib, maplib (disk), Jena, RDF4J, Kolibrie on native macOS. oxigraph and rdflib on Linux aarch64 VM. QLever, Virtuoso, GraphDB, Neo4j, and dotNetRDF via Docker. Server-based frameworks (QLever, Virtuoso, GraphDB, Neo4j) have query times that include HTTP round-trip overhead of ~0.5–1 ms; dotNetRDF runs in-process inside the container with no network overhead.
